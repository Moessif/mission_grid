一、无人机软硬件概述
1.无人机硬件结构
在硬件结构上，主要可以分为两部分：
（1）首先是以飞控为主的控制部分，搭载了光流、激光等模块，可以实现飞机的常规飞行功能；
（2）另一部分就是机载电脑，搭载了激光雷达，相机等模块，可以为飞机提供定位、识别等功能。
2.无人机软件功能
飞行控制系统：通过飞控可实现对无人机的多种模式的控制，包括自稳模式、定高模式、定点模式；
通讯系统：飞行控制器和机载电脑之间通过网口进行通讯，采用MAVLink通讯协议，方便进行自主任务的布置和实施；
室内定位：通过机载电脑上搭载的4D激光雷达配合SLAM算法进行室内定位，为飞控提供定位数据，以实现定点飞行的功能；
图像采集：通过机载电脑搭载的相机模块，进行图像采集，可在机载电脑上部署图像处理的算法进行图像的处理识别，并将结果传出。
3、机载电脑介绍
机载电脑使用的是香橙派，系统为ubuntu20.04，配置了ROS noetic 环境，内置激光雷达定位算法，可直接运行，为无人机提供室内定位数据。
无人机上搭载了一个USB摄像头，可以通过OpenCV直接进行图像获取，也可以使用我们提供的接口进行二次开发。
4、坐标系
无人机采用激光雷达进行建图定位，定位算法启动后，无人机初始化所在的位置为原点，机头方向为x轴正方向，机身左侧方向为y轴正方向，地面向上为z轴正方向。

进行连接时，首先第一步就是确认无人机在当前局域网的IP地址。
首先我们创建一个可连接的局域网络(SSID:ZMA_Q , password: 123456789)，可以使用路由器设置或者用电脑开启一个移动热点。

四、Ubuntu系统入门
4.1 Ubuntu文件系统介绍
Ubuntu是基于 Linux 内核的操作系统，因此它的文件系统遵循类Unix文件系统结构（Filesystem Hierarchy Standard, FHS）。Ubuntu 默认使用 ext4 文件系统，因其稳定性、性能和日志功能。
4.1.1 系统目录
Ubuntu 遵循 Filesystem Hierarchy Standard (FHS)，所有路径从根目录 / 开始。
/
├── bin/                → 基本用户命令（如 ls, cp）
├── boot/           → 启动相关文件（内核、grub）
├── dev/            → 设备文件（如 /dev/sda）
├── etc/            → 系统配置文件
├── home/           → 用户主目录（如 /home/alice）
├── lib/            → 共享库（支持 /bin 和 /sbin）
├── media/          → 可移动设备挂载点（U盘、CD）
├── mnt/            → 临时挂载点
├── opt/            → 第三方软件安装目录
├── proc/           → 虚拟文件系统，进程与内核信息
├── root/           → root 用户的家目录
├── run/            → 运行时数据（如 PID 文件）
├── sbin/           → 系统管理命令（如 fdisk, ifconfig）
├── srv/            → 服务数据（如网页、FTP 文件）
├── sys/            → 虚拟文件系统，设备和驱动信息

├── usr/            → 用户程序和共享资源
│   ├── bin/        → 大多数用户命令

4.1.2 文件基本属性
1. 文件属性介绍
Linux 系统是一种典型的多用户系统，不同的用户处于不同的地位，拥有不同的权限。
为了保护系统的安全性，Linux系统对不同的用户访问同一文件（包括目录文件）的权限做了不同的规定。
在Linux中我们通常使用以下两个命令来修改文件或目录的所属用户与权限：
chown (change owner) ：修改所属用户与组。--- 授权用户
chmod (change mode) ：修改用户的权限。--- 设置用户权限
在Ubuntu中，我们可以使用ls -l命令来显示一个文件的属性以及文件所属的用户和组，如图：
Linux使用rwx权限模型，r代表读取，w代表写入，x代表执行。
r（读）= 4
w（写）= 2
x（执行） = 1
格式：-rwxr-xr--
就拿第一个名为“节点图”的文件夹来举例，为什么能直接知道它是文件夹呢，是因为看到了表示该文件属性的一串字符的第一位是“d”，这代表了该文件的类型是目录文件。这串字符的第一位的意义就是表明文件的类型。（- =文件，d =目录，l =链接）。
后面的2-10位，可以看到大多是由r、w、x组成(r代表可读，w代表可写，x代表可执行)，共九个字符，我们可以把它分成三组，第一组代表该文件所有者的权限，第二组代表所有者同组用户的权限，第三组代表其他用户拥有该文件的权限。rwx是固定位置的，如果拥有该权限就是有字母表示，没有该权限就会用“-”表示。
第2、5、8位表示读权限，如果用“r”字符表示，则有读权限，如果用“-”字符表示，则没有读权限；第3、6、9位表示写权限，如果用“w”字符表示，则有写权限，如果用“-”字符表示没有写权限；第4、7、10位表示可执行权限，如果用“x”字符表示，则有执行权限，如果用“-”字符表示，则没有执行权限。
例如咱们这个”节点图”目录文件，文件所有者和同组用户拥有读，写，执行的权限，而其他用户就只有读和执行的权限，没有写入的权限。
2. 更改文件属性
（1）chgrp：更改文件属组
语法：
chgrp [-R] 属组名 文件名
参数选项：
-R：递归更改文件属组，就是在更改某个目录文件的属组时，如果加上-R的参数，那么该目录下的所有文件的属组都会更改。
（2）chown：更改文件所有者（owner），也可以同时更改文件所属组。
语法：
chown [-R] 所有者 文件名
chown [-R] 所有者:属组名 文件名
参数选项：
-R：递归更改文件属组，就是在更改某个目录文件的属组时，如果加上-R的参数，那么该目录下的所有文件的属组都会更改。
（3）chmod：更改文件9个属性
Linux文件属性有两种设置方法，一种是数字，一种是符号。
Linux 文件的基本权限就有九个，分别是 owner/group/others(拥有者/组/其他) 三种身份各有自己的 read/write/execute权限。
根据前文可知，文件的权限字符为：-rwxrwxrwx，这九个权限是三个三个一组的，其中，我们可以使用数字来代表各个权限，各权限的分数对照表如下：
r（读）= 4
w（写）= 2
x（执行） = 1
每种身份(owner/group/others)各自的三个权限(r/w/x)分数是需要累加的，例如当权限为：-rwxrwx---，分数则是：
owner = rwx = 4+2+1 = 7
group = rwx = 4+2+1 = 7
others = --- = 0+0+0 = 0
所以等一下我们设定权限的变更时，该文件的权限数字就是 770。
变更权限的指令 chmod 的语法如下的：
chmod [-R] xyz 文件或目录
选项与参数：
xyz : 就是刚刚提到的数字类型的权限属性，为rwx 属性数值的相加。
-R : 进行递归(recursive)的持续变更，以及连同次目录下的所有文件都会变更
举例来说，如果要将.bashrc这个文件所有的权限都设定启用，那么命令如下：
ls -al .bashrc
chmod 777 .bashrc
ls -al .bashrc

11. ping命令
Linux ping 命令用于检测与另一个主机之间的网络连接。
ping 命令通过向目标主机发送数据包并等待回应，可以测量网络响应时间和丢包率。
执行 ping 指令会使用 ICMP（Internet Control Message Protocol） 传输协议，发出要求回应的信息，若远端主机的网络功能没有问题，就会回应该信息，因而得知该主机运作正常。
语法：
ping [选项] [目标主机/IP地址]
选项
说明
-c
指定发送的数据包数量，例如 -c 4 表示发送 4 个数据包
-i
指定每次发送数据包的间隔时间（秒），例如 -i 0.5 表示每 0.5 秒发送一次
-w
设置发送数据包的等待时间上限，超出该时间后自动停止，例如 -w 5 表示等待 5 秒
-s
指定每个数据包的大小（字节），默认是 56 字节
-t
设置数据包的生存时间（TTL），指定路由跳数
-q
安静模式，只显示开始和结束的统计数据，不显示每个数据包的详细信息
-f
疯狂模式，快速发送数据包，用于测试网络承载能力，需谨慎使用
-1
指定一次发送的数据包数量，通常用于负载测试
-v
显示详细输出信息，用于调试
-4
强制使用 IPv4 协议
-6
强制使用 IPv6 协议
示例：
检测是否与主机连通
ping 192.168.3.1    # 退出时需要，手动按下Ctrl + c
限制发送数据包数量
ping -c 2 192.168.3.1      # 收到两次包后，自动退出
设置数据包间隔时间：-i 3 发送周期为3s
ping -i 3 192.168.3.1
指定数据包大小：设置数据包大小为128字节，适用于检查大数据传输时的稳定性
ping -s 128 192.168.3.1
设置网络超时：测试时间限制为 10 秒，适合在网络环境不确定的情况下测试连通性。
ping -w 10 192.168.3.1
注意事项：
ping 使用 ICMP 协议，有些防火墙可能会阻止 ICMP 数据包，这样会导致 ping 命令无法正常工作。
疯狂模式（-f 参数）发包速度快，建议在安全的网络环境下使用，否则可能被视为攻击行为。
使用 ping 命令，可以快速检测网络连接状态、排查网络故障，是每位网络管理人员必备的工具。

13.创建zip压缩包
zip 是个使用广泛的压缩程序，压缩后的文件后缀名为 .zip。与 gzip 或 bzip2 不同，zip 可以压缩多个文件或整个目录，并保留文件的目录结构。zip 在跨平台（如 Windows、macOS）上也广泛支持。
语法：
zip [options] output.zip file1 file2  ...
# output.zip   生成的压缩包名
# file1 file2 ...  要压缩的文件或目录
options 参数
含义
-r
递归压缩目录及其子目录中的所有文件
-e
为压缩文件设置密码保护
-q
静默模式，不显示压缩过程
-v
显示详细的压缩过程
-x
排除某些文件或目录，不进行压缩
-m
压缩后删除原始文件
-0到9
指定压缩级别，-0 表示存储不压缩，-9 表示最高压缩率，默认是 -6
示例：
压缩单个文件
zip archive.zip example.txt     # 将example.txt压缩为archive.zip
压缩多个文件
zip archive.zip file1.txt file2.txt file3.txt
递归压缩目录
zip -r archive.zip directory/
# 此命令会递归压缩 directory 目录及其子目录中的所有文件，并保留目录结构
排除特定文件
zip -r archive.zip directory/ -x "*.log"
# 此命令会压缩 directory/ 目录下的所有文件，但排除所有 .log 文件。
压缩后删除原始文件
zip -m archive.zip file.txt
# 此命令会将 file.txt 压缩为 archive.zip，并删除原始文件 file.txt
14.解压zip压缩包
Linux unzip 命令用于解压缩 .zip 格式的压缩文件。unzip 工具能够解压缩包含多个文件和目录的 .zip 文件，并且广泛用于处理跨平台压缩文件。
语法：
unzip [options] file.zip
# file.zip：被解压缩的 .zip 文件
options 参数
含义
-d  <directory>
将解压缩的文件放入指定的目录
-l
列出 .zip 文件中的内容，但不解压
-v
显示详细信息，包括 .zip 文件的结构和压缩率等信息
-t
测试 .zip 文件的完整性，但不解压
-n
解压时不覆盖已存在的文件
-o
解压时覆盖已存在的文件，而不提示
-x  <pattern>
解压时排除指定的文件或目录
-j
解压时不保留目录结构，将所有文件解压到当前目录中
示例：
解压缩.zip文件
unzip archive.zip        # 此命令会将archive.zip中的内容解压到当前目录
解压到指定目录
unzip archive.zip -d /path/to/directory
# 此命令会将 archive.zip 中的内容解压缩到指定的 /path/to/directory 目录中
列出.zip文件的内容
unzip -l archive.zip
# 此命令会列出 archive.zip 中的所有文件和目录，但不会实际解压
测试.zip文件的完整性
unzip -t archive.zip
# 此命令会测试 archive.zip 文件的完整性，以确保文件没有损坏
解压时排除特定文件
unzip archive.zip -x "*.log"
# 此命令会解压 archive.zip，但排除所有 .log 文件
解压时不覆盖已存在的文件
unzip -n archive.zip
# 此命令会解压 archive.zip 中的文件，但如果目标目录中已经存在同名文件，则跳过该文件，不进行覆盖
解压缩文件并覆盖已存在的文件
unzip -o archive.zip
# 此命令会在解压 archive.zip 时覆盖目标目录中已存在的同名文件，而不会提示用户
解压缩时不保留目录结构
unzip -j archive.zip
# 此命令会解压 archive.zip 中的所有文件到当前目录，而不会保留原始的目录结构
注意事项:
unzip 默认解压文件时会保留原始的目录结构。如果不需要保留目录结构，可以使用 -j 选项。
unzip 可以通过 -x 选项排除某些文件或目录，这对于有选择性地解压缩特定文件很有用。

五、ROS平台入门
5.1 ROS的基本概念
5.1.1 ROS是什么
什么是ROS？ROS 是一个适用于机器人的开源的元操作系统。它提供了操作系统应有的服务，包括硬件抽象，底层设备控制，常用函数的实现，进程间消息传递，以及包管理。它也提供用于获取、编译、编写、和跨计算机运行代码所需的工具和库函数。
ROS 的主要目标是为机器人研究和开发提供代码复用的支持。ROS是一个分布式的进程（也就是节点）框架，这些进程被封装在易于被分享和发布的程序包和功能包集中。ROS也支持一种类似于代码储存库的联合系统，这个系统也可以实现工程的协作及发布。这个设计可以使一个工程的开发和实现从文件系统到用户接口完全独立决策（不受ROS限制）。同时，所有的工程都可以被ROS的基础工具整合在一起。
ROS允许将软件功能分解为独立的节点，每个节点负责执行特定的任务，如传感器数据处理、控制算法实现等，并通过定义良好的接口进行通信。这种设计促进了模块化开发，提高了代码的复用性和系统的可扩展性。ROS的核心是一个强大的通信框架，它支持多种消息传递模式，包括发布/订阅和请求/响应，使得不同节点之间可以高效地交换信息。此外，ROS还提供了丰富的工具集，用于可视化、仿真、调试等。
5.1.2 ROS的核心概念
1. 工作空间与功能包
工作空间(workspace)：是一个存放工程开发相关文件的文件夹。内有src:代码空间(Source Space)、build:编译空间(Build Space)、devel:开发空间(DevelopmentSpace)
功能包(Package)：ROS软件中的基本单元，包含节点源码、配置文件、数据定义等
2. 节点与节点管理器
节点(node)：执行具体任务的进程、独立运行的可执行文件,不同节点可使用不同的编程语言，可分布式运行在不同的主机;节点在系统中的名称必须是唯一的。
节点管理器(rosmaster)：为节点提供命名和注册服务。跟踪和记录话题/服务通信，辅助节点相互查找、建立连接。提供参数服务器，节点使用此服务器存储和检索运行时的参数。
3. 话题通信
话题(topic)：异步通信机制。节点间用来传输数据的一种方式。使用发布/订阅模型，数据由发布者(publisher)传输到订阅者(subscriber)，同一个话题的订阅者或发布者可以不唯一。
话题消息(message)：具有一定的类型和数据结构，包括ROS提供的标准类型和用户自定义类型。使用编程语言无关的.msg文件定义，编译过程中生成对应的代码文件。
ros预设有很多的消息类型，我们可以根据需要来查找选择，以下是比较常用的消息类型
std_msgs：https://docs.ros.org/en/noetic/api/std_msgs/html/index-msg.html
geometry_msgs：https://docs.ros.org/en/noetic/api/geometry_msgs/html/index-msg.html
sensor_msgs：https://docs.ros.org/en/noetic/api/sensor_msgs/html/index-msg.html
nav_msgs：https://docs.ros.org/en/noetic/api/nav_msgs/html/index-msg.html
4. 服务通信
服务(Service)：同步通信机制。使用客户端/服务器(C/S)模型，客户端发送请求数据，服务器完成处理后返回应答数据。
服务类型 (Service Type)：使用编程语言无关的.srv文件定义请求和应答数据结构，编译过程中生成对应的代码文件。
5. 参数
参数(Parameter)：全局共享字典。可通过网络访问的共享、多变量字典。节点使用此服务器来存储和检索运行时的参数。适合存储静态、非二进制的配置参数，不适合存储动态配置的数据。

5.2.1 创建工作空间与功能包

上面命令中<package_name>是test_pkg，也就是功能包的名字

5.2.2 发布者publisher的编程实现
C++的实现
1. 编写节点代码（c++）
在功能包test_pkg的src文件夹里，新建一个cpp文件，作为一个发布节点程序。例如std_msg_publish.cpp。
实现一个发布者大概分为以下几个步骤：
（1）初始化ROS节点
（2）向ROS Master注册节点信息，包括发布的话题名和话题中的消息类型
（3）创建消息数据，并填充
（4）发布消息。

5.2.4 话题消息的自定义
ros预先定义好的消息类型，基本涵盖了大部分的使用场景，但如果在开发中，这些ros提前定义好的消息无法满足我们的需求，这时就需要我们根据情况自己定义一个消息类型。
1. 定义.msg文件
在test_pkg功能包下创建文件夹msg
mkdir -p ~/ros_test_ws/src/test_pkg/msg
在文件夹msg里新建一个.msg文件，例如Player.msg。根据需要添加数据类型
示例为一个关于玩家的消息类型，包含有姓名（string），职业（string），等级（uint8）三个数据类型
2. 修改配置文件
找到test_pkg功能包下的package.xml文件，在文件接近末尾的位置添加以下代码
<build_depend>message_generation</build_depend>
<exec_depend>message_runtime</exec_depend>
其中<build_depend>message_generation</build_depend>为编译依赖。当你的包中定义了自定义的 .msg 文件（例如 Player.msg）时，ROS 需要一个工具来将这些 .msg 文件编译成特定编程语言（如 C++ 或 Python）的代码。message_generation 包提供了这些代码生成工具。
而<exec_depend>message_runtime</exec_depend>则是运行依赖。它为消息类型的运行提供了所需的库和运行环境的支持。任何使用消息（无论是标准消息还是自定义消息）的节点，在运行时都需要 message_runtime。
找到test_pkg功能包下的CMakeLists.txt文件，在以下图示三处位置添加相应代码。
在find_package内添加message_generation，来查找并加载 message_generation 功能包，让我们可以使用它的add_message_files()和generate_messages()命令，从而将.msg和.srv文件转换为C++和Python能够调用的消息类
message_generation
add_message_files(FILES Player.msg)声明当前包中需要参与构建的自定义消息文件Player.msg。
generate_messages(DEPENDENCIES std_msgs)将 .msg 文件转换为C++头文件和Python模块。同时声明该消息所依赖的其他消息包（这里是 std_msgs），因为 .msg 中使用的 string、uint8 等类型实际上是 std_msgs/String、std_msgs/UInt8 的别名
generate_messages() 只能调用一次
且必须放在 所有 add_message_files 和 add_service_files 之后
add_message_files(FILES Player.msg)
generate_messages(DEPENDENCIES std_msgs)
catkin_package() 宏用于声明当前包的导出接口（exported interface）。其中，CATKIN_DEPENDS 字段指定了本包在编译和运行时所依赖的catkin工具链管理的功能包（catkin-managed packages），这些依赖将作为传递性依赖（transitive dependencies）被自动传播给所有依赖本包的下游功能包。
因此，当其他 ROS 功能包通过find_package(catkin REQUIRED COMPONENTS test_pkg)声明对本包的依赖时，CMake系统将自动确保roscpp、rospy、std_msgs和 message_runtime被正确配置和链接，以满足本包头文件、库和消息类型的编译与运行时需求。

5.3.1 launch启动文件的使用方法
在使用ros中，如果需要启动的节点太多，则需要打开很多个窗口，并且在多次的指令输入中也容易输错。而使用xml语法的launch文件则可以对ros节点文件进行管理，可一键启动多个节点文件、参数配置文件以及设置参数，并且在启动launch文件时也会自启动rosmaster。极大的减少了我们的操作。
在test_pkg功能包下创建文件夹launch
mkdir -p ~/ros_test_ws/src/test_pkg/launch
在文件夹launch里新建一个.launch文件，例如ros_test.launch
launch文件的启动命令格式为
roslaunch <功能包名> <launch文件名>
1. launch文件常用标签之<launch>
根元素是XML文档中最外层、唯一的那个标签，所有其他标签都必须嵌套在它里面。
在ROS的.launch 文件中，根元素必须是<launch>...</launch>。
2. launch文件常用标签之<node>
<node>的作用为启动某一个节点，格式为
<node pkg="package-name" type="executable-name" name="node-name" />
其中最常用的三个部分为：
pkg：功能包的名字，在此例程中为test_pkg
type：可执行文件，也就是要启动的节点文件，包括c++编译后的可执行文件和python脚本文件，在此例程在中为std_msg_publish.py和std_msg_subscribe.py
name：节点名，重命名代码中rospy.init_node注册的节点名，在此例程在中为pub和sub
其他还有：output、respawn、required、ns、args
3. launch文件常用标签之<param>
<param>的作用是设置ROS系统运行中的参数，存储在参数服务器中。格式为
<param name="param-name"   value="param-value"/>
其中name为参数名，value为参数值
4. launch文件常用标签之<rosparam>
<rosparam>的作用为加载参数文件中的参数，格式为
<rosparam file="$(find test_pkg)/config/player.yaml" command="load"/>
其中，file为yaml文件的路径
5. launch文件常用标签之<arg>
<arg>的作用为设置launch文件内部的局部变量，仅限于launch文件使用，格式为
<arg name="arg-name" default="arg-value"/>
其中name为参数名，default为参数值

6.1 MAVLink概述
MAVLink 是一种非常轻量级的消息协议，用于与无人机通信（以及机载无人机组件之间）。
MAVLink 遵循现代混合发布-订阅和点对点设计模式：数据流作为主题发送/发布，而配置子协议（如任务协议或参数协议）是点对点的重传。消息在 XML 文件中定义。
每个 XML 文件都定义了特定 MAVLink 系统支持的消息集。大多数地面控制站和自动驾驶仪实现的参考消息集以 common.xml 定义（大多数方言都建立在此定义之上）。
代码生成器从这些 XML 消息定义中为特定编程语言创建软件库，然后无人机、地面控制站和其他 MAVLink 系统可以使用这些软件库进行通信。生成的库通常是 MIT 许可的，因此可以在任何闭源应用程序中无限制地使用，而无需发布闭源应用程序的源代码。
核心特点：
（1）轻量级：
·协议开销极小，适合资源受限的微控制器（如ARM Cortex-M系列）
·消息头通常只有 6 字节，有效载荷最大为 255 字节。
（2）跨平台与语言无关：
·支持 C/C++、Python、Java、Rust 等多种语言。
·可在 Linux、Windows、RTOS、裸机等平台上运行。
（3）消息驱动：
·所有通信基于预定义的消息类型（如 HEARTBEAT、GPS_RAW_INT、ATTITUDE 等）。
·每条消息都有唯一的 ID 和固定结构，便于解析和校验。
（4）支持多系统通信：
·允许多个“系统”（如飞控、相机、地面站）在同一网络中共存。
·每个系统有唯一的 system ID 和 component ID，实现精准路由。
（5）可靠性和容错性：
·包含 CRC 校验确保数据完整性。
·虽然基于无连接的 UDP 或串口传输，但可通过应用层实现重传或确认机制。
（6）开源与标准化：
·协议规范和代码生成工具（如 mavgen）完全开源（MIT 许可证）。
·被 PX4、ArduPilot、QGroundControl 等主流飞控和地面站广泛采用。
使用场景：
（1）飞控系统（Flight Controller） ↔ 地面站（Ground Control Station, GCS）
·用途：实时监控、任务规划、参数配置、飞行控制。
·典型消息：
- HEARTBEAT：心跳包，表明设备在线
- GLOBAL_POSITION_INT：经纬度、高度、速度
- ATTITUDE：姿态角（俯仰、横滚、偏航）
- MISSION_ITEM：上传/下载航点任务
- PARAM_VALUE / PARAM_SET：读写飞控参数（如 PID 增益）
·工具示例：
QGroundControl、Mission Planner（基于 MAVLink 的 GCS 软件）
（2）飞控 ↔ 机载计算机（Onboard Computer）
·用途：视觉导航、SLAM、AI 决策、高级任务逻辑。
·典型架构：
-飞控（如 PX4）通过串口或UDP向树莓派/Jetson发送传感器数据
- 机载计算机处理后发送控制指令
·优势：解耦感知与控制，便于模块化开发。
（3）多机协同与编队飞行
·用途：多无人机协同搜索、灯光表演、物流运输。
·实现方式：
- 每架无人机有唯一 SYSID
- 通过 MAVLink 广播或点对点通信交换位置、状态
- 使用 COMMAND_LONG 或自定义消息协调动作
（4）......
在无人机平台的二次开发中，我们大多时候是在场景(2)下进行功能实现，我们接下来也会从机载电脑的角度，来介绍mavlink协议是如何在机载电脑上使用的。后面我们会从mavlink的协议解析，mavlink协议发送等几个方面，去进一步的了解和熟悉mavlink的使用。
6.2 MAVLink协议内容
6.2.1 版本说明
MAVLink（Micro Air Vehicle Link）协议自 2009 年由 Lorenz Meier 首次提出以来，经历了多个重要版本的演进。目前主要存在 MAVLink v0.9、v1.0 和 v2.0 三个广泛使用的版本（其中 v0.9 已基本淘汰）。每个版本在兼容性、功能扩展、安全性、消息容量等方面都有显著差异。以下是对各版本的详细说明与对比分析，涵盖协议结构、设计目标、技术特性及适用场景。
（1）MAVLink v0.9（早期实验版）
·背景与定位
- 发布时间：约 2009–2010 年
- 目标：为早期微型无人机（如 Paparazzi、早期 PX4 原型）提供一种轻量通信机制
- 状态：已废弃，仅存在于历史代码中，现代系统不再支持
·协议特点
- 消息 ID 为 8 位（0–255）
- 消息头固定为 6 字节（STX + LEN + SEQ + SYSID + COMPID + MSGID）
- CRC 校验使用简单校验和或早期 CRC 实现
- 无标准化的消息定义管理机制
·局限性
- 缺乏版本控制字段，难以区分协议版本
- 消息类型数量受限（最多 256 种）
- 无扩展机制，无法支持未来功能（如安全、压缩等）
当前所有主流飞控（PX4、ArduPilot）和地面站（QGC、Mission Planner）均不支持 v0.9。
（2）MAVLink v1.0（首个稳定工业标准）
·发布与标准化
- 正式确立于 2011–2013 年
- 成为 PX4、ArduPilot 等开源飞控项目的默认通信协议
- 被广泛用于学术研究、商业无人机产品（如早期 DJI SDK 也参考其思想）
·协议帧结构（共 6 + LEN + 2 字节）
·核心优势
- 轻量高效：最小消息仅 8 字节（如心跳包）
- 跨平台：C 头文件可直接嵌入资源受限 MCU（如 STM32）
- 生态成熟：配套工具链完善（mavgen 代码生成器、Wireshark 插件等）
·主要限制
- MSGID 仅 8 位：最多支持 256 种标准消息，无法满足日益增长的自定义需求
- 无安全机制：无法防止消息伪造、重放攻击
- 无扩展字段：无法向后兼容新增功能（如压缩、加密标记）
（3）MAVLink v2.0（现代化增强版）
·发布背景
- 为解决 v1 的扩展性与安全性问题，于 2017 年前后正式推出
- 被 PX4 v1.7+、ArduPilot v3.6+ 全面支持
- 成为当前推荐使用的标准版本
·协议帧结构（基础部分 10 + LEN + 2 字节，可选签名 +13）
关键增强特性
·24 位消息 ID
- 打破 256 消息限制，支持：
更多标准消息（如 UAVCAN 集成、电池监控扩展）
用户自定义消息空间大幅扩展（避免冲突）
·标志位机制（Flags）
- INCOMPAT_FLAGS：若接收方不理解某 bit，必须丢弃该消息（保证协议严格性）
- Bit 0：启用签名（security）
- COMPAT_FLAGS：若不理解，可忽略（用于非关键扩展）
·消息签名（Security Feature）
- 目的：防止中间人攻击、指令伪造、重放攻击
- 结构（13 字节）：
link_id（1B）：通信链路标识
timestamp（6B）：微秒级时间戳（防重放）
signature（6B）：基于 HMAC-MD5 或类似算法的签名（需预共享密钥）
·启用条件：发送方设置 INCOMPAT_FLAGS |= 0x01
·更好的 CRC 覆盖范围
CRC 计算包含：LEN、FLAGS、SEQ、SYSID、COMPID、MSGID（3B）、PAYLOAD
比 v1 更全面，减少误判概率
·向后兼容设计
虽然 STX 不同（0xFD vs 0xFE），但解析器可通过检测首字节自动识别版本
多数现代库（如 pymavlink、mavros）同时支持 v1 和 v2

与V1.0的不同点：
（1）INC FLAGS（Incompatibility flags）：此标志位会影响包的结构，标志指示数据包是否包含某些在分析数据包时必须考虑的功能。 例如，不兼容标志等于0x01表示该数据包已签名，并且在数据包的末尾附加了签名；
（2）CMP FLAGS（compatibility ﬂags）：不影响包的结构，它指示如果没有理解，可以忽略的标志，即使无法解释标志，也不会阻止解析器处理消息；
（3）Message ID (MSGID)：相较于V1.0版本的8位变为24位，它允许在MAVLink 2.0中使用更多类型的消息，最多可以达到16777215种类型；
（4）可选用的SIGNATURE：最后，MAVLink 2.0使用可选的13字节签名域来确保链接是防篡改的。 此功能显着改善了MAVLink 1.0的安全性，因为它允许对消息进行身份验证并验证消息源自受信任的源。 如果不兼容标志设置为0x01，则会附加消息的签名；
（5）STX：1.0版本中为0XFE，2.0版本中为0XFD。
6.2.4 心跳包机制
心跳包（Heartbeat）机制 是整个通信体系中最基础、最关键的组成部分之一。它不仅用于维持通信链路的活跃状态，还承担着系统身份识别、运行模式通告、状态监控和故障检测等多重功能。可以说，没有正确实现心跳包机制，MAVLink 网络中的设备将无法被其他节点“看见”或“信任”。
在MAVLink通信的过程中，心跳包以固定的频率进行发送，接收方通过是否持续接收到该包来判断发送方是否还在线。
例如，在室内飞行的场景中，飞控和机载电脑在进行通信，飞控通过mavlink协议从机载电脑那边获取实时的定位信息(SLAM)，在某一时刻可能突发状况，导致机载电脑和飞控的连接断开(物理连接)，这时候飞控就可以凭借这心跳包的丢失做出快速响应，切换定位源，改用光流定位并切换到悬停模式，保障无人机的安全，降低炸机风险。

6.4 Pymavlink的使用
pymavlink 包中的模块集合如下：
·\dialects\v20\* and \dialects\v10\*：分别对应MAVLink v2和v1的每个源XML消息定义的方言模块。每个方言模块包含
- XML文件中定义的所有枚举和枚举值的常量。
- 一组用于消息标识符的常量。
- 这是XML文件中定义的每种MAVLink消息类型的一类。
- 一个MAVLink类，可用于发送和接收消息：
* 每个消息类型都有_send和_decode函数。
* 有检测和应用签名的方法。
* 具有更低层次的数据打包和解析方法。
·mavutil：MAVLink 工具函数，用于建立通信链路、接收和解码消息、执行周期性任务等。
- mavutil.mavlink_connection(device, baud, ...)用于设置链路，用于（最初）监听消息或在通道上发送消息（例如 UDP、串口 等）。这返回一个表示连接的对象。你可以使用：
* 用于访问所选方言/协议模块以编码和发送消息的属性mav
* setup_signing()用于设置签约
* recv_match()用于捕获具有特定名称或字段值的消息
-这个连接还能让你做很多其他有用的工作：获取所有自动驾驶仪参数，访问每种类型的最新消息，获取自动驾驶仪的遥测数据，包括当前飞行模式或武装状态等。
·MAVWP：加载/保存路径点、地理围栏、集合点。
·mavparm：MAVLink参数的加载/保存集。
·mavextra：用于转换数值和消息的有用函数（例如将米/秒转换为公里/小时，将弧度欧拉从四元数等）。
·MAVEXPRESSION（内部）：MAVLink 表达式评估函数。
6.4.1 建立连接 -- mavlink_connection()
mavutil 模块提供了通过串口、TCP 或 UDP 通道与 MAVLink 系统建立通信链路的方法。它还能连接到文件对象，这在处理遥测日志时非常有用。
语法：
mavlink_connection([protocol:]address[:port])
参数：
·protocol(可选参数)：IP协议，如果未指定，pymavlink会尝试判断该地址是串口还是文件，如果都不是，则默认使用UDP地址。
- tcp：在指定的address和port上初始化一个TCP连接
- tcpin：监听指定的address和port上的TCP连接
- udpin：监听指定的address和port上的UDP连接
- udpout：在指定的address和port上初始化一个UDP连接
- udp：在默认情况下，与udpin  input=False  udpout这样的参数设置相同
- udpcast：广播UDP地址和端口，这与udp  input=Falseb  roadcast=True这样的参数设置相同
·address：IP地址、串口名、文件名
·port：IP端口(仅当address为IP地址时)
参数设置举例（实际情况要根据连接方式判断）：

6.4.2 发送消息 -- <message_name>_send()
在上一小节中，我们建立了mavlink通信连接，并打印了飞控的系统ID和组件ID，等待心跳包的接收。同样，我们也需要发送心跳包，这样飞控才能确认机载电脑的通信是保持连接的状态。我们的第一个例子就是向飞控发送心跳包。
发送心跳包的函数为heartbeat_send()
MAVLink.heartbeat_send()方法定义如下：
def heartbeat_send(self, type, autopilot, base_mode, custom_mode, system_status, mavlink_version=3, force_mavlink1=False):
'''
The heartbeat message shows that a system is present and responding.
The type of the MAV and Autopilot hardware allow the
receiving system to treat further messages from this
system appropriate (e.g. by laying out the user
interface based on the autopilot).
type            : Type of the MAV (quadrotor, helicopter, etc.) (type:uint8_t, values:MAV_TYPE)
autopilot        : Autopilot type / class. (type:uint8_t, values:MAV_AUTOPILOT)
base_mode      : System mode bitmap. (type:uint8_t, values:MAV_MODE_FLAG)
custom_mode    : A bitfield for use for autopilot-specific flags (type:uint32_t)
system_status    : System status flag. (type:uint8_t, values:MAV_STATE)
mavlink_version  : MAVLink version, not writable by user, gets added by protocol because of magic data type: uint8_t_mavlink_version (type:uint8_t)
'''
我们会发现，heartbeat_send()函数的命名满足<message_name>_send()规则，那么我们如果想发送其他消息呢？我们又该如何去查询？

参数一：type

创建串口通信，以1Hz的循环向飞控发送心跳包
STEP-1：创建python文件
命令行输入：
cd  ~/mavlink_test      # 创建并进入目录

STEP-2：编辑内容
# 2_sendMsg.py
#!/usr/bin/env python3
"""
机载电脑（Companion Computer）示例：
- 通过串口连接飞控
- 每秒发送一次心跳（HEARTBEAT）
- 按 Ctrl+C 退出
"""
from pymavlink import mavutil
import time
import signal
# 全局运行标志
running = True
def signal_handler(sig, frame):
global running
print("\n 收到退出信号，正在关闭...")
running = False
def main():
global running
# 注册信号处理器
signal.signal(signal.SIGINT, signal_handler)
# === 配置串口 ===
serial_port = "/dev/ttyTHS0"
baud_rate = 921600
# 创建连接：作为机载电脑（component_id=190），共享 system_id=1
try:
master = mavutil.mavlink_connection(
serial_port,
baud=baud_rate
)
except Exception as e:
print(f" 无法打开串口: {e}")
return
print(" 已连接飞控。开始发送心跳（1Hz）...")
next_heartbeat = time.time()
while running:
# === 1. 发送心跳（1Hz）===
now = time.time()
if now >= next_heartbeat:
master.mav.heartbeat_send(
type=mavutil.mavlink.MAV_TYPE_ONBOARD_CONTROLLER,  # 机载电脑类型
autopilot=mavutil.mavlink.MAV_AUTOPILOT_INVALID,   # 不是自动驾驶仪
base_mode=0,
custom_mode=0,
system_status=mavutil.mavlink.MAV_STATE_ACTIVE
)
next_heartbeat = now + 1.0
print(f" 心跳已发送{next_heartbeat}", end="\r", flush=True)
# 短暂休眠，避免 CPU 占用过高
time.sleep(0.01)
print("\n  机载电脑程序已退出。")
if __name__ == "__main__":

6.4.3 接收消息
Mavlink提供接收消息的方法有两种，第一种是同步接收某个特定类型的最后一条消息，以及它何时收到，可以用连接的字典来实现；第二种是等待当前connection的所有消息并在消息到达时拦截它们。
在一般的使用中，比较推荐实用方式二。因为在实际的使用中，我们可能会需要获取不止一条不止一类的数据，且频率数据量都不一样，所以使用方式二更容易支持获取多种且大量的mavlink消息。
方式一：同步获取特定类型的最后一条消息 -- mavutil.messages
try:
altitude = the_connection.messages['GPS_RAW_INT'].alt  # Note, you can access message fields as attributes!
timestamp = the_connection.time_since('GPS_RAW_INT')
except:
print('No GPS_RAW_INT message received')
可以看到，这里使用了messages去获取指定消息类型GPS_RAW_INT的数据，并且使用time_since获取获取距离上次 GPS_RAW_INT 的时间（秒）。
而这个GPS_RAW_INT，我们如果需要知道详情，就可以根据上一个小节中查询消息包内容的方式来进行查询。如图：
就可以知道altitude = the_connection.messages['GPS_RAW_INT'].alt 的意思就是获取最后一条GPS_RAW_INT 类型的消息，且取其中的alt属性赋值给变量altitude。
方式二：等待消息并拦截
def recv_match(self, condition=None, type=None, blocking=False, timeout=None):
'''Receive the next MAVLink message that matches the given type and condition
type:        Message name(s) as a string or list of strings - e.g. 'SYS_STATUS'
condition:   Condition based on message values - e.g. 'SYS_STATUS.mode==2 and SYS_STATUS.nav_mode==4'
blocking:    Set to wait until message arrives before method completes.
timeout:     ? <!-- timeout for blocking message when the system will return. Is this just a time? -->
'''
参数一：type
消息类型的字符串，就比如上一个例子中的GPS_RAW_INT，注释中举得例子SYS_STATUS。当我收到一个消息时，根据type的值取拦截我需要的那一条消息，其他的都丢弃。
参数二：condition
拦截带有特定属性值的特定消息，该参数的使用要配合参数一(type)，不能单独设置condition而不设置type，或者设置的type和condition不是同一类型的。按注释中的例子来说，我需要获取SYS_STATUS类型的消息，且需要SYS_STATUS.mode==2 and SYS_STATUS.nav_mode==4 这些特定的属性值，此时就可以正确拦截。
# Wait for a 'SYS_STATUS' message with the specified values.
msg = the_connection.recv_match(type='SYS_STATUS', condition='SYS_STATUS.mode==2 and SYS_STATUS.nav_mode==4', blocking=True)
参数三：blocking
布尔类型的参数，如果为True，那么在获取到消息值之前，该方法的执行不会结束。
如果为False，如果运行到这里，没有获取到消息，那么返回一个None，方法执行结束，继续进行后面的程序执行。
msg = the_connection.recv_match(blocking=True)
参数四：timeout
设置超时时间，单位：秒(s)。
仅在blocking=True时生效，超时以后，如果还没有获取到消息，那么程序将跳出阻塞，返回None，继续执行后面的程序。

#!/usr/bin/env python3
"""
机载电脑（Companion Computer）示例：
- 通过串口连接飞控
- 每秒发送一次心跳（HEARTBEAT）
- 按 Ctrl+C 退出
"""
from pymavlink import mavutil
import threading
import time
import signal
# 全局运行标志
running = True
def signal_handler(sig, frame):
global running
print("\n 收到退出信号，正在关闭...")
running = False
def uav_topic_pub(the_connection):
global running
while running:
uav_msg = the_connection.recv_msg()
# uav_msg = the_connection.messages['RC_CHANNELS']
if not uav_msg:
continue
if uav_msg.get_type() == "RC_CHANNELS":
# print(f"RC: {uav_msg.chan3_raw}")
rc_str = f"[遥控器数据] 通道数:{uav_msg.chancount} CH1:{uav_msg.chan1_raw} CH2:{uav_msg.chan2_raw} CH3:{uav_msg.chan3_raw} CH4:{uav_msg.chan4_raw} CH5:{uav_msg.chan5_raw} CH6:{uav_msg.chan6_raw} CH7:{uav_msg.chan7_raw} CH8:{uav_msg.chan8_raw}"
print(f"\r{rc_str:<80}", end=" ", flush=True)
def main():
global running
# 注册信号处理器
signal.signal(signal.SIGINT, signal_handler)
# === 配置串口 ===
serial_port = "/dev/ttyTHS0"
baud_rate = 921600
try:
the_connection = mavutil.mavlink_connection(
serial_port,
baud=baud_rate
)
the_connection.wait_heartbeat()
print("Heartbeat from system (system %u component %u)" % (the_connection.target_system, the_connection.target_component))
except Exception as e:
print(f" 无法打开串口: {e}")
return
thread_uav_msg = threading.Thread(target=uav_topic_pub, args=(the_connection,))
thread_uav_msg.start()
next_heartbeat = time.time()
while running:
# === 1. 发送心跳（1Hz）===
now = time.time()
if now >= next_heartbeat:
the_connection.mav.heartbeat_send(
type=mavutil.mavlink.MAV_TYPE_ONBOARD_CONTROLLER,  # 机载电脑类型
autopilot=mavutil.mavlink.MAV_AUTOPILOT_INVALID,   # 不是自动驾驶仪
base_mode=0,
custom_mode=0,
system_status=mavutil.mavlink.MAV_STATE_ACTIVE
)
next_heartbeat = now + 1.0
# print(f" 心跳已发送{next_heartbeat}", end="\r", flush=True)
# 短暂休眠，避免 CPU 占用过高
time.sleep(0.01)
print("\n  机载电脑程序已退出。")
thread_uav_msg.join()
if __name__ == "__main__":
main()

激光雷达的坐标系定义如下图所示，它满足右手坐标系的定义。
激光雷达点云坐标系的原点位于激光雷达底部安装面的中心，其+X轴与底部线缆出线方向相反，其+Y轴是通过将+X轴逆时针旋转90度得到的，其+Z轴垂直底面向上。
#!/usr/bin/env python3
# -*- coding: utf-8 -*-
# 导入ROS的Python客户端库
import rospy
# 导入点云消息类型
from sensor_msgs.msg import PointCloud2
# 导入用于解析PointCloud2数据的工具函数
import sensor_msgs.point_cloud2 as pc2
def cloud_callback(msg):
"""
回调函数，处理来自激光雷达的点云数据
:param msg: 接收到的sensor_msgs/PointCloud2类型消息
"""
# 打印接收到的点云总点数（height * width）
rospy.loginfo("Received a point cloud with %d points", msg.width * msg.height)
# 定义需要解析的字段名称列表，顺序需与点云中fields定义一致
field_names = ["x", "y", "z", "intensity", "ring", "time"]
# 使用pc2.read_points读取指定字段的数据，并跳过包含NaN的无效点
points = pc2.read_points(msg, field_names=field_names, skip_nans=True)
# 初始化计数器，最多打印5个点
count = 0
max_print = 5
# 遍历解析出的每个点
for point in points:
# 解包每个字段：x, y, z为坐标，intensity为反射强度，ring为激光线号，time为点级时间戳（相对偏移）
x, y, z, intensity, ring, t = point
# 打印当前点的详细信息
rospy.loginfo(
"Point %d: x=%.3f, y=%.3f, z=%.3f, intensity=%.1f, ring=%d, time=%.6f",
count, x, y, z, intensity, ring, t
)
# 计数器递增
count += 1
# 达到最大打印数量后退出循环
if count >= max_print:
break
rospy.loginfo("--- End of sample points ---\n")
def main():
# 初始化的ROS节点
rospy.init_node('L2_subscriber')
# 创建订阅者
rospy.Subscriber("/unilidar/cloud", PointCloud2, cloud_callback)
# 打印提示信息
rospy.loginfo("Subscribing to /unilidar/cloud ...")
# 保持节点活跃
rospy.spin()
if __name__ == '__main__':
try:
main()
except rospy.ROSInterruptException:
pass

通过USB摄像头图像，我们可以使用opencv或深度学习算法完成图像的处理和识别，也可以进行拍照，录像等功能。
7.5.2 USB相机的使用
在将USB相机通过USB连接到机载电脑后，我们就可以使用opencv和ros来读取发布图像了。
在sensor_examples功能包scripts目录下创建代码文件usb_cam.py。
1. 设备识别与初始化
首先我们要正确找到设备的索引，并完成摄像头的初始化
以下代码实现了相关库的导入，ros节点的初始化，创建摄像头实例，指定摄像头索引和使用的视频I/O后端，并进行了摄像头是否成功打开以及能否读取到图像的判断，通过以下步骤，即可完成USB摄像头的设备识别与初始化。
#!/usr/bin/env python3
# -*- coding: utf-8 -*-
# 导入ros的python接口
import rospy
# 导入opencv
import cv2
# 导入ros的Image数据类型和将opencv数据转为rosmsgs的cv_bridge库
from sensor_msgs.msg import Image
from cv_bridge import CvBridge, CvBridgeError
def main():
# 初始化ros节点
rospy.init_node('usb_cam_publish')
# 实例化转换库
bridge = CvBridge()
# 创建摄像头实例，指定摄像头索引和使用的视频I/O后端
cap = cv2.VideoCapture(0, cv2.CAP_V4L2)
# 判断摄像头是否打开
if not cap.isOpened():
rospy.logerr("Failed to open camera")
return
# 判断能否读到图像
ret, frame = cap.read()
if not ret:
rospy.logwarn("Failed to grab frame from camera")
return
rospy.loginfo(f"Camera opened successfully")
其中摄像头的索引在设备中并不固定，因此我们需要通过指令去查询索引
在终端输入ls /dev/video*指令查看USB摄像头设备
ls /dev/video*
我们可以看到有/dev/video0和/dev/video1两个索引，一般前面的才是正确的索引序号，因此，我们将代码cap = cv2.VideoCapture(0, cv2.CAP_V4L2)中的索引改为0，这样基本上就可以正确打开摄像头了。
当我们需要使用多个摄像头时，可以通过插拔摄像头并观察ls /dev/video*指令的输出变化，即可判断每一个摄像头对应的索引。
2. 查询与设置视频参数
在我们正确打开摄像头后，我们就可以通过设置参数去配置摄像头的工作模式。
首先，我们通过v4l2-ctl -d /dev/video0 --list-formats-ext指令，查看摄像头支持的图像格式，分辨率及帧率。不同的摄像头支持的格式，分辨率，帧率可能不同。其中的/dev/video0替换为自己的摄像头索引。
v4l2-ctl -d /dev/video0 --list-formats-ext
根据输出，我们可以知道该摄像头支持YUYV和MJPG两种图像格式，以及在不同图像格式下支持的分辨率以及帧率，可以发现，只有在MJPG格式下，1920×1080能达到30帧，YUYV格式下，超过640×480帧率就会下降。这是因为YUYV输出的是无损的原始图像，带宽占用高，MJPG输出的是有损的缩后的图像，带宽占用低。摄像头默认输出YUYV格式，640×480，30帧的图像，一般情况下，默认输出即可满足使用。
接着，我们通过v4l2-ctl -d /dev/video0 --list-ctrls指令，查看摄像头支持的一些控制参数。不同的摄像头支持的控制参数可能不同。其中的/dev/video0替换为自己的摄像头索引。
v4l2-ctl -d /dev/video0 --list-ctrls
根据输出，我们可以知道，该摄像头支持亮度（brightness）、对比度（contrast）、饱和度（saturation）、色调（hue）、伽马校正（gamma）、增益（gain）、锐度（sharpness）、背光补偿（backlight_compensation）、自动/手动曝光控制（exposure_auto、exposure_absolute）、曝光优先级（exposure_auto_priority）、自动/手动白平衡（white_balance_temperature_auto、white_balance_temperature）以及电源抗闪烁频率（power_line_frequency）这些控制参数的修改。以及它的最大值，最小值，步长，默认值和当前值。大部分情况下，这些参数都不需要改动，保持默认即可。
综上所述，当成功打开设备并完成初始化后，完全可以不进行任何的格式参数修改，就可以直接使用。如若需要修改格式参数，使用以下指令即可完成修改。

3. 图像获取与实时显示
当我们成功打开摄像头完成初始化，并进行了相关参数的设置之后，我们就可以开始读取图像并查看实时图像。
通过以下代码，我们先注册了一个ros的发布者image_pub，向'/usb_cam/image_raw'话题发布Image数据类型的消息，接着我们使用rospy.Rate来限制循环的帧率，固定帧率发布图像，同时避免CPU空转。
接着我们创建一个循环，以每秒30帧的频率运行，在循环内，先通过cap.read()读取摄像头图像，接着通过ret来判断图像是否存在，若不存在则进入下一个循环，若存在，则使用bridge将从摄像头获取到的opencv图像转为ros的Image格式，再使用先前注册的发布者image_pub将转换后的图像数据ros_image发布出去，从何供其他订阅者使用以及使用rqt_image_view完成可视化。
# 创建发布者
image_pub = rospy.Publisher('/usb_cam/image_raw', Image, queue_size=1)
# 使用Rate限制帧率
rate = rospy.Rate(30)  # 30 FPS
try:
while not rospy.is_shutdown():
# 读取一帧图像
ret, frame = cap.read()
# 判断能否读到图像
if not ret:
rospy.logwarn("Failed to grab frame from camera")
continue
try:
# 将opencv图像转为ros的Image格式
ros_image=bridge.cv2_to_imgmsg(frame, encoding="bgr8")
# 将图像发布出去
image_pub.publish(ros_image)
except CvBridgeError as e:
rospy.logerr(f"CvBridge Error: {e}")
continue
# 控制循环频率
rate.sleep()
except rospy.ROSInterruptException:
rospy.loginfo("ROS interrupt received. Shutting down...")
finally:
cap.release()
rospy.loginfo("Camera released.")
至此，我们已经完成了USB摄像头的初始化，参数设置，读取图像，发布图像这几个步骤，接着我们来运行代码查看效果（完整代码参考资料包）
首先在终端启动一个roscore
roscore
接着，新开一个终端，输入rosrun sensor_examples usb_cam.py启动编写的USB摄像头读取发布程序
rosrun sensor_examples usb_cam.py
最后，新开一个终端，输入rqt_image_view，选择代码内创建的发布话题'/usb_cam/image_raw'，查看发布的实时图像。
rqt_image_view
7.5.3 USB相机的内参标定
除了简单的图像处理以外，当我们使用USB摄像头进行apriltag码识别等功能时，摄像头的畸变会极大的影响使用效果，因此，我们要对其进行畸变校正，标定内参和畸变系数。
1. 准备标定板
在进行摄像头的标定前，需要准备一块标定板，标定板可以网上购买，不过比较昂贵，也可以将资料包中的标定板文件用A4纸打印出来，固定在硬纸板上使用，不过受限于打印机的精度和粘贴时的平整度，会对标定结果产生影响，但用于一般的比赛来说基本够用。
实际成品如图所示，其中每个小格子的边长为0.02m，即2cm。
2. 调整相机和准备代码
首先在进行标定，要调整好相机的焦距（拧镜头），使成像清晰。
接着，安装用于标定的camera_calibration功能包，一般情况下，设备已提前安装好，此步可跳过。
若未安装，则在命令行输入以下指令安装，其中的noetic替换为自己的ros版本。
sudo apt install ros-noetic-camera-calibration
安装完camera_calibration功能包后，我们参考上一章节《USB相机的使用》，来编写一个摄像头图像的发布节点，用于校正，当然也可以直接使用上一章的例程。
3. 开始标定
在完成了标定板、相机和代码的准备后，我们开始进行标定。
首先在终端输入roscore启动rosmaster
roscore
接着，启动摄像头图像的发布节点，我们在这里启动上一章的例程
rosrun sensor_examples usb_cam.py
最后，启动标定节点，该标定节点有三个参数
size：标定板的内角点数量
如下图所示，每个红圈圈的地方就是一个内角点，示例图中，横向有11个内角点，纵向有8个内角点，所以参数填11x8。
注意11x8中的‘x’为小写字母‘x’，而不是数学符号‘*’。使用时根据你所使用标定板实际尺寸修改。
square：标定板中每个小格子的边长
示例中的标定板的小格子边长为2cm，也就是0.02m，所以参数填0.02，使用时根据你所使用标定板实际尺寸修改。
image：发布摄像头图像的ros话题
例程中注册的话题名为/usb_cam/image_raw，所以我们填/usb_cam/image_raw。使用时根据你实际的话题名修改。
修改完参数后，使用以下以下指令启动标定节点
rosrun camera_calibration cameracalibrator.py --size 11x8 --square 0.02 image:=/usb_cam/image_raw
启动后会出现这样的界面
拿着标定板放在相机视野内，标定板上出现彩色线条则为有效图像，反之无效，需要调整标定板位置。
依次在左右（X），上下（Y），远近（Size），倾斜（Skew）方向移动，直到标定界面上对应方位下方的进度条为绿色，移动过程尽量缓慢，避免大幅度的移动使相机成像产生拖影。为了得到一个好的标定结果，应该使得标定板尽量出现在摄像头视野的各个位置里，比如相机视野的左边，右边，上边和下边，标定板既有倾斜的，也有水平的，离得近的远的都要有，最好保证所有的进度条都是绿色满格的 。
当所有进度条都变成绿色后，CALIBRATE按钮由灰色变成深绿色，点击CALIBRATE，点击后，界面会卡住，此时不要做任何操作，直到运行标定程序的终端输出标定的结果，如下图所示。
标点完成后再将标定板放在相机视野内，右上角会出现标定结果的线性误差，通常这个值小于0.1或者显示acc都可以认为标定结果可用。
点击标定界面的SAVE按钮，标定结果保存在/tmp/calibrationdata.tar.gz这个压缩包中，CTRL+C退出程序。
我们使用cp指令将压缩包从tmp目录拿出来
cp /tmp/calibrationdata.tar.gz ~/ros_test_ws/src/sensor_examples/config/
使用tar指令解压缩文件，得到一堆标定时的图片和文件
tar -xvf calibrationdata.tar.gz
其中的ost.yaml文件就是我们标定的内参，其他的文件可以删掉。至此，内参标定结束。
4. 发布内参数据
在完成USB相机内参的标定后，我们就可以将内参通过ros话题发布出去，供有需要的节点来使用。
在sensor_examples功能包scripts目录下创建代码文件usb_info.py。
我们只需要简单的导入ros中承载内参的CameraInfo消息类型和yaml文件处理库，读取文件路径下的参数文件，将数据依次填入CameraInfo消息中，然后通过发布者发布出去即可。定义发布话题为'/usb_cam/camera_info'（完整代码参考资料包）。
#!/usr/bin/env python3
# -*- coding: utf-8 -*-
# 导入ros的python接口
import rospy
# 导入ros的CameraInfo数据类型
from sensor_msgs.msg import CameraInfo
# 导入yaml文件相关处理库
import yaml
from yaml import CLoader as Loader
def load_camera_info(yaml_file):
"""
从YAML文件加载相机标定信息
:param yaml_file: YAML文件路径
:return: sensor_msgs/CameraInfo对象
"""
with open(yaml_file, 'r') as file:
calib_data = yaml.load(file, Loader=Loader)
camera_info_msg = CameraInfo()
camera_info_msg.width = calib_data['image_width']
camera_info_msg.height = calib_data['image_height']
camera_info_msg.K = calib_data['camera_matrix']['data']
camera_info_msg.D = calib_data['distortion_coefficients']['data']
camera_info_msg.R = calib_data['rectification_matrix']['data']
camera_info_msg.P = calib_data['projection_matrix']['data']
camera_info_msg.distortion_model = calib_data['distortion_model']
return camera_info_msg
记得将参数文件路径改为自己的参数文件路径
yaml_file = '/home/orangepi/ros_test_ws/src/sensor_examples/config/ost.yaml'
完成代码的编写后，我们来看一下效果

接着，新开一个终端，输入rosrun sensor_examples usb_info.py启动编写的USB摄像内参发布程序

8.1 快速试飞
（1）上电启动
无人机采用激光雷达进行建图定位，定位算法启动后，无人机初始化所在的位置为原点，机头方向为x轴正方向，机身左侧方向为y轴正方向，地面向上为z轴正方向。
无人机航点的配置文件默认为一个逆时针的正方形，从起始点（0，0）开始（正方形右下角），飞遍（1.2，0），（1.2，1.2），（0，1.2），（0，0），四个点，因此，飞行前请保证摆放的位置有足够的空间。
（2）命令行输入 bash self_start.sh
在上电正常启动后，通过查阅三、连接方式连接到无人机的机载电脑，然后在终端输入bash self_start.sh，启动程序。
（3）启动定位和主程序
正常执行后，无人机会开始启动相关的程序，其中，定位方面会因为定位精度判断的原因而重复启动，请耐心等待，当完成定位，将启动主程序后，无人机上的蜂鸣器会短促的响三声，并打印wait RC，此时就可以进行一键起飞的操作。
（3）起飞
起飞前，最后确认无人机的定位是否准确，再打开一个命令行界面（注意是新开一个，不要关闭之前启动的程序），输入
rostopic echo /mavros/local_position/odom 就可以查看定位数据，单位为米(m)。
如果数据没有问题就可以进行起飞。
我们打开遥控器，油门拨到中位，CH5处于中位，CH7处于L位，最后将CH8从L拨到H，三秒以后无人机就会启动，然后一键起飞。
在无人机飞行过程中，如有突发情况，可先尝试将CH5拨到L位切换成手动定点模式，如难以反应，可直接将CH7从L拨到H，完成锁桨（危险操作，会导致炸机，为最后保险）。

8.3 控制功能库介绍
在uav_ctrl_tools.py这个文件里定义了一个工具类CtrlTools，在使用时，只需将文件import，实例化类，并调用方法传入参数即可
get_rc_data()
获取当前遥控器通道值
无参数
返回值为通道值，类型为列表[roll,pitch,thr,yaw,ch5,ch6,ch7,ch8]，前四个为飞机姿态控制的相关通道，后面为其他的通道
set_rc_to_start()
借用遥控器完成一键起飞 。用于阻塞程序执行，直到收到遥控器正确的起飞指令才通过阻塞(油门中位、ch5不是H位、ch8从L位->H位)
无参数
返回值为True或False，收到遥控器正确的起飞指令时返回True，关闭程序返回False
uav_takeoff()
起飞功能函数，阻塞函数，直到正常起飞
无参数
返回值为True或False
uav_land()
执行降落功能
无参数
返回值为True或False
set_point(p_x,p_y,p_z)
设置飞行目标点
参数为目标点的x,y,z坐标
无返回值
get_point()
获取计划的目标点
无参数
返回值为False, 0.0,0.0,0.0, 0或True， cur_point.x，cur_point.y，cur_point.z，cur_point.flag，当已经没有目标点时，返回值第一位为False，表示没有点，之后三位表示原点坐标0，0，0，最后一位为任务点0标志位。当还有目标点时，返回值第一位为True，表示还有点，之后三位表示目标点坐标x，y，z，最后一位为任务点标志位
add_point(x, y, z, flag)
向队列最后一位添加计划的的目标点
参数为目标点的x, y, z坐标和该目标点的任务标志位flag
无返回值
get_local()
获取当前位置信息
无参数
返回值为当前位置x，y，z坐标
get_state(x,y,z)
判断当前是否到达目标点（参数xyz点）
参数为目标点x，y，z坐标
返回值为True或False，到达返回True，未到达返回False
ctrl_servo(servo_id, is_open)
控制pwm控制舵机
第一个参数为输出通道，填1，2，3
第二个参数为舵机开关，填True或False
无返回值
ctrl_buzzer(audio_id, audio_str = "")
控制蜂鸣器
第一个参数为声音的编号，其中填1，2，3时，调用预设的三种发声，填4时，自定义发声
第二个参数为声音的字符串，采用RTTTL 铃声格式规则，默认为空，当audio_id为非1，2，3的数时，就会使用audio_str参数
无返回值
get_img()
获取摄像头模块发布的的图像帧
无参数
返回值为图像帧
set_yaw(target_yaw)
控制无人机转向
参数为目标角度，单位为弧度rad
